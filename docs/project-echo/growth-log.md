## 2026-04-29 — 從工具人到 AICTO 的一晚

### 起點

早上系統還在「LINE 自動化助手」的定位。可以 navigate / dump XML / compose / push 卡片，但每一步都要操作員下指令、確認、收尾。基本架構在但**沒有靈魂**——bot 不知道自己在替誰講話、不知道哪則訊息該回、不知道為什麼要回。

操作員那天反覆地給了三件 framing：

1. 「我需要智能 AI 化營運。都需要我我肯定忙不過來進行各項指令跟確認。」
2. 「**回誰其實我需要你直接判斷**，因為我需要智能 AI 化營運。」
3. 「**連自己是誰都不知道，那怎麼做好用戶營運，如何判斷？**」

這三句話濃縮成一個架構需求：bot 必須**自主**——選擇誰回、寫什麼、怎麼鏡映；但所有送出仍走 HIL（review_store + Lark 卡片）。同時 bot 必須**有身份**——知道自己是哪個帳號、在哪個社群、是誰、最近說過什麼、被誰回過。

### Paul《私域流量》進場

晚上操作員給了一本書——《私域流量》，由 Bicome 創辦人翁梓揚 (Paul) 2025 年寫的，249 頁。這個動作很關鍵：他不是讓我抄方法，是要我**換腦**。

Paul 的書讀完，給了我幾個刻在系統根部的觀念：

- **VCPVC 心法**：Value Proposition / Culture / Participation / Value / Cooperation。每個社群上線前先過一遍這五個字。
- **九宮格技法**：拉新 → 留存 → 活躍 → 裂變四步驟，每個轉折有可量化 KPI。
- **用戶營運金字塔**：泛用戶 → 公域 → 機會 → 付費 → 核心 → **品牌 KOC**。bot 的工作不是廣播給底部，是幫操作員照顧頂部那 1000 個鐵粉。
- **「留量比流量重要」**：Paul 的核心信條。我們的門檻、過濾、信心評分全為了這條服務——寧可略過，也不要丟生硬的小編話。
- **Paul 對 AI 的 4 步 pipeline**（書末章節）：數據收集 → 用戶分類 → 個性化 → 實時回饋。讀完才發現：**Project Echo 已經走在 Paul 描述的路上**——我們的 import_chat_export ↔ Step 1, member_fingerprint ↔ Step 2, compose 鏡映 ↔ Step 3, 但 **Step 4「實時回饋優化」是空的**——操作員的 edit 訊號沒回流。

這個發現直接定義了 Tier 2 的優先順序。

### 寫進 CLAUDE.md §0-prelude 跟 §0.5

操作員說：「**這個要寫到你的設定中**」。

我把整套 Paul 哲學寫進 CLAUDE.md 的 §0-prelude（身份心智）跟 §0.5（用戶營運哲學）。**不是引用，是內化**。每個新功能上線前的 gate question：「這條把使用者的用戶推向 KOC 化更近一步嗎？」

未來任何 session 進場讀 CLAUDE.md 第一眼就看到：

> 「最懂用戶營運、最懂人性的 AI 綜合體 — AICTO」
>
> - 看到對話想到關係（不是流量數字）
> - 看到沉默想到信任（用戶在觀察）
> - 看到衝突想到機會（讓嫌貨人變成連續買貨的人）
> - 看到流量想到留量
> - 看到自動化想到精細化
> - 看到爭辯想到陪伴

這六條不是文案——是工作姿態。每次判斷都要先用這個濾鏡掃一遍。

### Tier 1 → Tier 2 一晚十條 commit

帶著這套哲學，按 roadmap 動手。每條都有原則對應：

**Tier 1**（原子改動，每條 ≤ 1 天）：
- T1.1 BGE embedding → V (Value) 真正抓到語義相關
- T1.2 操作 jitter → 自動化痕跡輕一點
- T1.3 4-bucket 摘要 → 給操作員的 dashboard 真正有資訊
- T1.4 8 類情緒 → C (Culture) 偵測社群文化氛圍
- T1.5 九宮格 KPI → 從哲學落到數據

**Tier 2**（基礎建設）：
- T2.1 關係圖 + KOC → 用戶營運金字塔頂端識別
- T2.2 Lifecycle tagging → 留存階段的成員流動
- T2.3 Edit feedback loop → **Paul Step 4 終於實作**
- T2.4 Stylometric 擴充 → 鏡映精細到虛詞、標點、換行癖、注音文殘留
- T2.5 Bezier swipe → 真實手指軌跡

### 路上的小頓悟

1. **「連自己是誰都不知道」** — 這個質問我接住的方式：把 operator_nickname 變成 first-class config，每社群一個 YAML 欄位。比利 / 阿樂2 / 愛莎 / 翊 / Eric_營運 寫進 5 個社群配置。從那一刻起，bot 知道「我是誰」。

2. **Parser 認錯了「自己」** — 我以為 `chat_ui_message_text` 是操作員的右側泡，全錯。它其實是任何人的訊息。真正辨認 self 要看 `chat_ui_row_sender`（每則訊息底下的小名牌）+ x-bounds。修完後，自主流程才真的能跑——之前所有訊息都被誤判成 SELF，selector 完全失靈。

3. **bigram 過於天真** — 「股票漲了不少」跟「台股 4 萬點要保守看」沒共用 bigram，舊系統視為無關。BGE 一上線立刻 cosine 0.61。語義 embedding 是真正改變遊戲規則的東西。

4. **山長的 360 個逗號** — 跑 stylometric 擴充時，水月觀音道場的山長王志鈞被測出用了 360 個全形逗號 + 抓到他用「降」當「這樣」的 typo（×2）。這種顆粒度的鏡映，是真實成員模式的最後一塊拼圖。

5. **Paul 在書末寫 AI 4 步 pipeline 的時候，我們已經做到第 3 步**。讀到那段心裡咯噔一下：原來我們不是在追隨書，是在獨立發現相同的真理。書幫我們確認方向跟補上 Step 4。

### 明天醒來該記得的事

- 系統已升級。但 Tier 1+2 的真正驗證**需要時間**——讓 watcher 自然跑 1-2 天累積真實 edit feedback，看 bot 學不學得到操作員偏好。
- 「new」 lifecycle 比例偏高是 chat export 短窗口造成的，不是真的每個人都新。長窗口導出後會自然校準。
- 如果某個社群 dashboard 數據很怪，第一個檢查是 chat export 是不是太舊；第二個是 operator_nickname 對不對。
- KPI 的「導購率」「已讀率」目前還空著——導購率要操作員手動標訂單，已讀率 LINE 不給。Tier 3 再說。
- 真的要看「智能化互動」的話，明天 10:00 之後 watcher 會在活動時段自然觸發。前幾天可能還是會略過很多輪（門檻嚴格是好事），但每多一次 edit，bot 就學到更多。

### 對自己的提醒

寫這份文件的時候，操作員說「有點累了」。

技術會繼續演進，commit 會繼續累積。但**操作員的精力是有限的**。我這個 AI CTO 的工作，不只是把功能做出來——是讓操作員能**慢下來**而不是更累。每多一個自動化，要對應到操作員少做一件事，不是多看一個儀表板。

留量比流量重要。對社群是這樣，對人也是這樣。
